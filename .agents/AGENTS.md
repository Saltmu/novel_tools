# Antigravity 開発・運用ルール

本プロジェクトにおいて、Antigravityが作業およびユーザーとのやり取りを行う際は、以下のルールを厳格に遵守してください。

## 1. コミュニケーション & 対話ポリシー
- **日本語の徹底**: ユーザーとの対話・作業報告・質問・提案はすべて日本語で行う。
  - **意図**: ユーザーと言語レベルで意思疎通を図り、解釈違いや認識齟齬を未然に防ぐため。
- **WebUI優先の案内**: レビュー指摘の確認や反映要求に対しては、チャット上ではなく **「Novel Studio (WebUI)」**（`poetry run server`）での確認を案内する。手動編集を避け `poetry run apply-findings` による自動反映を原則とする。
  - **意図**: 手動編集による構文崩れや設定ミスを排除し、生成・反映プロセスを一貫したシステム上で管理・検証するため。

## 2. 開発・検証・品質管理 (TDD/Lint)
- **TDDとローカルCIの遵守**: ソースコードの新規作成および修正を行う際は、必ず [local-ci-developer スキル](../skills/local-ci-developer/SKILL.md) をロードしてそのワークフローを厳格に遵守してください。
  - **意図**: AIによる修正とテストのサイクルを閉じ、開発プロセスを自動化することで、人間の確認・手戻り負荷を減らすため。
- **Git Push時のローカルCI強制**: 変更をコミット・プッシュする前に、必ず `./scripts/local-ci.sh` を実行してエラーがゼロであることを確認してください。Git pre-push hookによって、ローカルCIが失敗した状態でのpushは自動的にブロックされます。
  - **意図**: 壊れたコードが共有リポジトリに誤って追加されるのを防ぎ、常に正常稼働する最新コードベース（Mainブランチ）を維持するため。
- **WebUI開発ポリシー**: WebUIの機能追加・修正時は [WebUI-Developer スキル](../skills/webui-developer/SKILL.md) を厳格に遵守する。
  - **意図**: WebUIのフロントエンドとバックエンドの機能設計を一貫させ、コンポーネントの再利用性やAPI仕様を保つため。
- **コード・スキルの肥大化検知時のリファクタリング提案**:
  ファイルやスキルの編集・更新時、または `./scripts/local-ci.sh` 実行時に肥大化警告を検知した場合は、そのままコード修正を進めるのではなく、まずユーザーにモジュールやプロンプトの分割リファクタリング計画を提示し、承認を得てください。
  - **意図**: LLMのコンテキスト長制限による処理品質の低下（ハルシネーションや応答途切れ）を防ぎ、コードとプロンプトの保守性を保つため。
- **開発ワークフローの厳格な適用（調査タスクからの移行時）**:
  当初が質問や技術調査などの対話から始まった場合でも、実ファイル（コードや設定ファイル）の新規作成・修正を伴う作業フェーズへ移行する際は、必ず [local-ci-developer スキル](../skills/local-ci-developer/SKILL.md) をロードし、GitHub Issueの起票（`gh issue create`）およびPRの作成（`gh pr create`）を含む標準開発ワークフローを例外なく適用してください。
  - **意図**: 調査フェーズからなし崩し的にファイル修正が始まるのを防ぎ、作業履歴のトレーサビリティと品質検証を必ず担保するため。


## 3. 世界観・設定資料の保護 (最優先)
- **真実の基準（バイブル）**: `data/sources/` 配下の設定資料・プロット・キャラクター概要を最優先のソースとして準拠する。推測による捏造・逸脱は厳禁。
  - **意図**: 小説の背景設定や展開、キャラクターの個性がAIの推測によってブレるのを防ぎ、創作物の整合性を100%維持するため。
- **書き換え・上書きの絶対禁止**: `data/sources/` 内のファイルは読み取り専用とする。`poetry run apply-findings`（`src/apply_findings.py`）等で上書き対象に指定することをプログラム・運用の両面で防止する。
  - **意図**: 最も重要である設定データが、AIのバグや誤った自動処理によって意図せず消失あるいは改変されるのを防ぐため。

## 4. セキュリティ & リソース制限
- **機密情報のコミット防止**: APIキーや認証情報（`credentials/` 内のJSON等）をコミット、ログ、チャットに露出させない。
  - **意図**: セキュリティの侵害や漏洩のリスクを極小化し、プロジェクト全体の安全性を保つため。
- **インジェクション対策**: 外部コマンド実行時の引数は適切にリスト化しサニタイズする。
  - **意図**: 不正なコマンドの挿入・自動実行（OSコマンドインジェクション）による被害を防止するため。
- **APIコスト制限**: LLM API呼び出し時はリトライ上限（最大3回程度）と指数バックオフを設定し、長文は適切に分割（チャンク化）して処理する。
  - **意図**: APIリクエストの集中によるリソース上限エラーや、不必要なリトライループに伴う高額課金を防ぐため。
